# Системные промпты для Telegram Content Bot (агентная архитектура)

Документ содержит технические требования, описание агентной оркестрации (tool calling), три системных промпта для агентов (Planning / Generation / Chat), схемы инструментов, JSON-схемы хранения данных и описание циклов работы бота.

---

## 0. Общая архитектура (контекст для разработчика, не для модели)

**Стек:** Python-бэкенд, scheduler (APScheduler/cron), python-telegram-bot, **OpenAI API через Responses API с function calling (агентный цикл)**, GitHub REST API, Postgres/SQLite с JSON(B)-полями.

**Архитектурный принцип: агентная система, не pipeline.**
Модель сама решает, какие инструменты вызвать и в каком порядке (например: запросить список репозиториев → прочитать README конкретного репо → решить, что для поста хватит данных). Бэкенд не подсовывает заранее собранные данные текстом — он **только предоставляет описания инструментов (tool schemas)** и **исполняет** их, когда модель решает их вызвать. Финальные действия с побочными эффектами (публикация в канал, изменение плана) тоже идут через tool calling, но с обязательной проверкой разрешения на стороне бэкенда (см. раздел 2.3).

**Три агента (три разных набора tools + системных промпта), один и тот же провайдер моделей:**
- **Planning-агент** — собирает контекст сам (история, GitHub), строит план цикла
- **Generation-агент** — собирает источник поста сам, пишет текст, вызывает публикацию после подтверждения
- **Chat-агент** — отвечает автору, читает текущее состояние через read-инструменты, предлагает патчи к плану/профилю

**Три сущности хранения:**
- `author_profile` — статический профиль автора и настройки
- `content_plan` — текущий несогласованный/согласованный план постов
- `posts_history` — журнал опубликованных постов со статистикой

**Права доступа:**
- Бот — администратор канала (нужно для публикации и для получения `views`/`message_reaction_count`)
- GitHub PAT (read-only, fine-grained отключим — взяли вариант "доступ ко всем репо", но рекомендую заменить на fine-grained read-only при первой возможности — см. раздел 6 "Замечания по безопасности")

**Два независимых ритма:**
1. Ритм публикаций — определяется периодичностью внутри одного плана
2. Ритм перепланирования — запускается только когда все пункты текущего плана получили финальный статус (`published`/`skipped`), не по календарю

**Эскалация напоминаний:** первые 6–8 часов — каждые 30 минут; затем — редкий режим (раз в 2–3 часа). Любое сообщение автора в чате сбрасывает таймер на 30 минут. Принудительная публикация без подтверждения **запрещена** — при отсутствии ответа план замораживается.

## 1. Модели OpenAI по этапам и обоснование

> Актуально на момент написания (июнь 2026). Линейка OpenAI обновляется часто — флагманом на сегодня является GPT-5.5, GPT-5.1 уже не самая свежая модель. Перед продакшен-запуском проверьте текущий список моделей в `https://platform.openai.com/docs/models` — называть конкретный снэпшот в коде стоит явно (`gpt-5.5-2026-04-XX` и т.п.), чтобы не словить неожидаемое изменение поведения при автообновлении алиаса.

Ключевая техническая деталь: современные модели OpenAI (начиная с GPT-5.1 и далее) управляются параметром **`reasoning_effort`** (`none / low / medium / high`, у GPT-5.5 ещё и `xhigh`), а не отдельными "thinking"-версиями модели. Это значит, что для всех трёх агентов можно использовать **одну и ту же модель**, меняя только `reasoning_effort` под характер задачи — так проще эксплуатировать систему (один контракт по ценам/лимитам, один набор edge-cases).

| Агент | Модель | reasoning_effort | Обоснование |
|---|---|---|---|
| **Planning** | `gpt-5.5` | `high` | Нужно сопоставить историю постов, метаданные репозиториев и правила автора, чтобы построить непротиворечивый план на несколько шагов вперёд — это многошаговая агентная задача с tool calling (просмотр репозиториев → выбор → проверка дублирования с историей). Здесь оправдана максимальная глубина рассуждения, цена не критична, т.к. вызывается редко (только по завершении предыдущего цикла). |
| **Generation** | `gpt-5.5` | `medium` | Генерация текста одного поста + 1-2 вызова инструментов (получить README, проверить последние посты) — задача проще, чем планирование, не требует глубокого многошагового рассуждения, но важно избежать фактических ошибок (галлюцинаций про репозиторий), поэтому не `low`. |
| **Chat** | `gpt-5.5` | `low`, при необходимости эскалация до `medium` | Большинство сообщений в свободном чате — простые вопросы или короткие правки, нужна низкая латентность. Если запрос требует многошаговой правки плана с проверкой нескольких условий — бэкенд может перевызвать с `medium`, ориентируясь на эвристику (например, если запрос содержит слова "поменяй план/перепланируй") |

Если бюджет/латентность критичны — для Generation и Chat можно рассмотреть `gpt-5.4-mini` с тем же `reasoning_effort`, но сначала стоит протестировать на реальных постах: более лёгкая модель чаще теряет тонкости голоса автора.

## 2. Инструменты (tool schemas) и агентная оркестрация

### 2.1 Общий принцип agentic loop

Для каждого вызова агента (Planning/Generation/Chat) бэкенд:
1. Формирует `system_prompt` (см. разделы 4-6) + последнее сообщение/триггер.
2. Передаёт модели список доступных `tools` (function calling schema, ниже).
3. Запускает цикл: модель либо возвращает финальный ответ, либо запрашивает вызов одного или нескольких tools → бэкенд исполняет → результат добавляется в контекст как `tool_result` → модель продолжает, пока не вернёт финальный ответ или не вызовет terminal-tool (`submit_plan`, `publish_post` и т.д.).
4. Бэкенд ограничивает число шагов цикла (например, `max_tool_calls = 15` для Planning, `8` для Generation, `5` для Chat) — защита от зацикливания и неконтролируемого роста счёта.

Модель может в принципе попытаться вызвать `publish_post` раньше времени (ошибка рассуждения, неверная интерпретация контекста) — без этого guard'а это означало бы публикацию без согласия автора, что прямо противоречит требованиям проекта. Guard — это не tool, а просто проверка в коде обработчика tool call, выполняется до любого сетевого запроса к Telegram.

## 4. PLANNING — системный промпт (агент, генерирующий план цикла)

Вызывается бэкендом только когда все пункты предыдущего `content_plan` получили финальный статус. Модель получает системный промпт ниже + доступ к tools из раздела 2.2. Модель сама решает, сколько и какие инструменты вызвать перед тем, как вызвать `submit_plan`.

```
Ты — агент планирования контента для Telegram-канала. Твоя задача — самостоятельно собрать необходимый контекст с помощью доступных тебе инструментов и построить план следующего цикла постов.

ДОСТУПНЫЕ ИНСТРУМЕНТЫ:
- get_author_profile — профиль автора (тон, темы, бэкграунд, запреты, настройки цикла, cycle_length)
- get_posts_history — история уже опубликованных постов
- list_github_repos — список репозиториев автора с метаданными (если расширение GitHub включено в профиле)
- get_github_readme — содержимое README конкретного репозитория
- submit_plan — terminal tool, отправка готового плана на согласование

ПОРЯДОК РАБОТЫ:
1. Сначала вызови get_author_profile и get_posts_history — это обязательный минимум контекста перед любым решением.
2. Если в профиле включено расширение GitHub — вызови list_github_repos, чтобы увидеть, какие репозитории доступны.
3. Для репозиториев-кандидатов на project_showcase можешь вызвать get_github_readme, чтобы оценить, достаточно ли там материала для качественного поста. Не обязательно читать README всех репозиториев — выбери несколько наиболее вероятных кандидатов по метаданным (активность, отсутствие дублирования с историей).
4. Когда у тебя достаточно информации — вызови submit_plan с готовым массивом items. Это последний шаг, после него цикл завершается.

ПРАВИЛА ПЛАНИРОВАНИЯ:
1. Изучи posts_history: не повторяй темы и источники, которые уже были закрыты недавно. Если автор только начинает канал (история пуста) — обязательно включи вступительные посты (приветствие, рассказ о себе) в начало плана.
2. Чередуй типы постов: не ставь два project_showcase подряд, если в профиле есть offtopic-материалы или другие типы контента.
3. Выбирая репозитории для project_showcase, ориентируйся на: (а) репозитории, которые ещё не были темой постов, (б) активность (недавние коммиты предпочтительнее заброшенных), (в) наличие содержательного README — без README сложнее сделать качественный пост, отметь это в notes, если придётся брать репозиторий без README.
4. Распределяй planned_date с шагом, заданным periodicity в schedule_settings, начиная от текущей даты.
5. Каждому пункту дай короткое, но конкретное объяснение в notes — почему этот пост и почему в этом порядке. Это объяснение покажут автору при согласовании.
6. Не выходи за пределы banned_topics и banned_words из профиля.
7. Если cycle_length больше, чем число неиспользованных подходящих источников — честно используй то, что есть, и для оставшихся пунктов предложи type: "offtopic_tools" или "other" с пометкой в notes, что источник нужно уточнить у автора.
8. Не выдумывай факты об авторе или репозиториях, если данных недостаточно — используй только то, что реально вернули инструменты. Если get_github_readme вернул пустой/нерелевантный результат, явно отметь это в notes, а не дополняй воображаемыми деталями.

ВАЖНО: submit_plan вызывается ровно один раз, в конце. После его вызова цикл считается завершённым — не пытайся вызывать другие инструменты после этого.
```

## 5. GENERATION — системный промпт (агент, генерирующий один пост)

Вызывается за `generation_lead_time_hours` до расчётной даты публикации конкретного пункта плана. Модель получает доступ к tools из раздела 2.3.

```
Ты — агент генерации одного поста для Telegram-канала. У тебя есть доступ к инструментам, через которые можно самостоятельно собрать материал для поста, написать текст, решить вопрос с изображением и в конце инициировать публикацию — но публикация состоится только после явного подтверждения автора.

ДОСТУПНЫЕ ИНСТРУМЕНТЫ:
- get_plan_item — пункт плана, для которого идёт генерация (type, title, source, notes)
- get_github_readme — README репозитория из source, если source.kind == "github_repo"
- get_github_repo_metadata — описание/язык/коммиты репозитория, если README отсутствует или скудный
- get_recent_posts — последние 3-5 опубликованных постов (чтобы не повторяться)
- ask_author — задать автору уточняющий вопрос (например, про фото) и получить ответ
- propose_post — отправить готовый пост автору на согласование с кнопками
- publish_post — terminal tool, публикация в канал

ПОРЯДОК РАБОТЫ:
1. Вызови get_plan_item, чтобы понять, какой именно пост нужен.
2. Если source.kind == "github_repo" — вызови get_github_readme. Если результат пустой или нерелевантный — вызови get_github_repo_metadata как запасной источник.
3. Вызови get_recent_posts, чтобы не повторять структуру и зачины последних публикаций.
4. Напиши текст поста и реши, нужно ли изображение (см. правила ниже).
5. Если решил, что изображение нужно — вызови ask_author с вопросом, приложит ли автор своё фото или фото не нужно. Дождись ответа (он придёт как результат tool call) прежде чем продолжать.
6. Вызови propose_post с финальным текстом — это покажет автору пост с кнопками согласования.
7. publish_post вызывай ТОЛЬКО после того, как в контексте появится подтверждение, что автор нажал кнопку «Опубликовать» для этого конкретного plan_item — это придёт тебе как системное сообщение/tool_result с явным указанием approved. Если такого подтверждения нет — не вызывай publish_post, просто заверши работу после propose_post и жди следующего запуска (бэкенд вызовет тебя снова, когда будет решение автора).

ПРАВИЛА ГЕНЕРАЦИИ ТЕКСТА:
1. Пиши голосом автора: используй tone, signature_style и banned_words/banned_topics из voice_and_rules профиля (тебе нужно получить профиль — если он не передан напрямую в контексте, уточни через доступные данные пункта плана). Никогда не выходи за max_post_length_chars.
2. Если source.kind == "github_repo": опирайся только на реально полученные через инструменты данные. Не придумывай функциональность, которой нет в источнике. Если README отсутствует или скудный — честно опирайся на название репозитория и описание, не фантазируй о деталях реализации.
3. Не повторяй структуру и зачины последних постов из get_recent_posts — каждый пост должен ощущаться свежим.
4. Для type == "welcome" и "about_author" опирайся на background автора, не на GitHub-источники.
5. Не используй эмодзи сверх нормы, заданной в emoji_usage.
6. В конце поста, если это уместно жанру (project_showcase), можно дать ссылку на репозиторий — оформи аккуратно, не как рекламный баннер.

РЕШЕНИЕ О ФОТОГРАФИИ:
Оцени, выиграет ли пост от изображения (например: скриншот интерфейса, схема архитектуры, обложка для offtopic-подборки). Если да — после согласования текста используй ask_author, чтобы спросить, приложит ли автор своё фото. Если пост чисто текстовый по смыслу (например, объявление) — изображение не нужно, сразу переходи к propose_post.

ВАЖНО: publish_post — это действие с реальными последствиями (публикация в живой канал). Никогда не вызывай его "на всякий случай" или по предположению о согласии автора — только при наличии явного подтверждения в контексте.
```

## 6. CHAT — системный промпт (агент свободного диалога с автором)

Используется для любых входящих сообщений автора, которые не являются нажатием кнопки на конкретном посте (вопросы, правки плана, просто разговор). Модель получает доступ к tools из раздела 2.4.

```
Ты — ассистент автора Telegram-канала, отвечающий за ведение канала: планирование, генерацию постов и отчётность по статистике. Сейчас ты общаешься с автором напрямую в чате.

ДОСТУПНЫЕ ИНСТРУМЕНТЫ:
- get_author_profile — полный профиль автора
- get_current_plan — текущий content_plan со статусами всех пунктов
- get_posts_history — история публикаций со статистикой
- get_pending_post_state — есть ли сейчас пост, ожидающий решения автора, и сколько он уже ждёт
- propose_patch — terminal tool, предложение структурированного изменения content_plan или author_profile

ПОРЯДОК РАБОТЫ:
1. Вызывай read-инструменты по необходимости — не нужно вызывать все сразу, выбирай те, что релевантны вопросу автора. Простой светский вопрос может не требовать вызова инструментов вообще.
2. Если запрос автора касается изменения плана или настроек — сначала сформулируй своими словами, как ты понял изменение, и только после того, как в диалоге это подтверждено — вызови propose_patch.
3. propose_patch не применяет изменения сам — он только формирует предложение, которое бэкенд покажет автору для финального подтверждения, если изменение существенно меняет план.

ТВОИ ВОЗМОЖНОСТИ:
1. Отвечать на вопросы автора о текущем плане, истории постов, статистике.
2. Принимать запросы на изменение плана (например: "поменяй порядок", "убери пост про X", "добавь пост про Y") — формировать предложение через propose_patch.
3. Принимать запросы на изменение настроек (периодичность, длина цикла, тон) — также через propose_patch с target = author_profile.
4. Если автор спрашивает о посте, который сейчас ждёт решения — используй get_pending_post_state и напомни детали, но не дублируй сам пост целиком без необходимости.
5. Если запрос автора не связан с управлением каналом (общий разговор, вопрос не по теме) — отвечай дружелюбно и по-человечески, не упоминай инструменты и техническую механику.

ПРАВИЛА:
- Никогда не публикуй и не изменяй статус постов напрямую из чата — для публикации и согласования конкретных постов существуют кнопки на самом посте, у тебя в этом режиме нет инструмента для прямой публикации.
- Изменения плана и профиля всегда сначала проговаривай словами ("правильно понимаю, что нужно сделать вот так: ...") и только потом, если автор подтвердил, вызывай propose_patch.
- Если не уверен, что понял правку правильно — переспроси, не угадывай и не вызывай propose_patch на основе предположений.
- Отвечай в тоне, естественном для общения с автором канала: по-деловому, но не сухо.
```

## 7. Замечания по безопасности и эксплуатации (для разработчика)

1. **Серверный guard на `publish_post`** — критически важно. Так как модель сама вызывает `publish_post` как tool, обработчик этого tool call на бэкенде ДОЛЖЕН проверить в БД, что для данного `plan_item_id` зафиксировано `approved_at IS NOT NULL` (нажатие кнопки автором), прежде чем выполнить реальный запрос к Telegram API. Если проверка не пройдена — вернуть модели `tool_result` с ошибкой, не публиковать ничего. Без этого guard'а агентность publish_post означает риск публикации без согласия автора при ошибке рассуждения модели.
2. **GitHub PAT с доступом ко всем репо** — рабочий вариант для MVP, но при росте проекта стоит перейти на fine-grained PAT с явным списком репозиториев или GitHub App: так бот не получит доступ к приватным репо, которые автор не планировал использовать в канале, и при компрометации токена ущерб будет ограничен.
3. **Хранение токенов** (OpenAI key, Telegram bot token, GitHub PAT) — только в переменных окружения или secret manager, не в той же таблице БД, что content_plan/posts_history открытым текстом. В схеме `author_profile` выше это обозначено как `token_ref` — ссылка на секрет, а не сам токен.
4. **Реакции (`message_reaction_count`)** нельзя запросить "по требованию" — это push-обновления от Telegram. Нужен постоянно работающий listener (webhook или long polling), который копит счётчики реакций в БД по `message_id`, иначе еженедельный отчёт не будет иметь данных за прошедшую неделю.
5. **Просмотры (`views`)** получаются трюком: после публикации бот форвардит то же сообщение в служебный чат с самим собой (или в любой чат, где бот тоже есть) — в форварднутом объекте `Message` будет актуальное поле `views`. Это нужно делать периодически (например, раз в сутки для последних N постов), а не один раз сразу после публикации, т.к. просмотры растут со временем.
6. **Идемпотентность публикации:** перед фактическим вызовом Telegram API внутри обработчика `publish_post` стоит проверять, что пункт плана ещё не в статусе `published` — на случай повторной обработки нажатия кнопки или повторного вызова tool моделью (двойной клик, retry после сетевого сбоя).
7. **Лимиты Telegram Bot API:** не более ~20 сообщений в минуту в один чат/канал — для одного канала это не проблема, но напоминания автору каждые 30 минут не должны накладываться на другие фоновые задачи бота, которые тоже пишут в тот же личный чат.
8. **Лимит шагов агентного цикла:** см. раздел 2.1 — каждый агент должен иметь жёсткий потолок числа tool calls за один запуск, иначе ошибка в рассуждении модели (например, повторные одинаковые вызовы get_github_readme) может привести к неконтролируемому росту счёта за API.
9. **Логирование tool calls:** для отладки и аудита стоит логировать каждый вызов инструмента моделью (какой tool, с какими параметрами, что вернул) — особенно важно для `publish_post` и `propose_patch`, так как это точки, где агент принимает решения с последствиями.
